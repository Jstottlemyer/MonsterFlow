# Spec Review — spec-qa-terminal-formatting

**Date:** 2026-05-16
**Reviewers:** gaps (opus) · scope (sonnet) · requirements (sonnet) · codex-adversary
**Gate mode:** permissive (frontmatter)
**Overall health:** Significant Gaps — **FAIL**

## TL;DR

Codex's adversarial pass exposed a load-bearing assumption that is unverified and likely wrong: **Claude Code probably does not render raw ANSI escape codes embedded in model-generated markdown**. The supported color mechanism is status-line stdout (real ESC bytes from shell scripts), not assistant-message text. The spec's claim that `commands/check.md` already uses raw ANSI in failure banners was fabricated — `grep -rn '\\033' commands/` returns zero matches across the entire `commands/` tree.

If the assumption is wrong, every AC fails: the test would pass (grep finds `\033[32m` in the source files) while the actual rendering shows literal `\033[32m**a)**\033[0m` text on screen — *worse* than the current all-default state, not better.

This spec cannot proceed to `/blueprint` without an empirical spike that verifies what Claude Code actually does when a model emits `\033[32m` in its response.

## Before You Build — 3 architectural blockers

### B1 (architectural · blocker · class:architectural)
**The core rendering assumption is unproven.** Codex finding #1: ANSI escape codes are supported for status-line stdout (real ESC bytes from shell scripts), not for arbitrary assistant-message Markdown generated by the model. The repo currently has zero `\033` occurrences in `commands/`, contradicting the spec's claimed precedent.

**Required before /blueprint:** a minimal spike in a throwaway file or session: add a one-line instruction asking Claude to emit `\033[32mTEST\033[0m`, run it interactively, capture what the user sees. Three possible outcomes:
- **(a) Colored** — assumption holds, spec proceeds with hardening
- **(b) Literal `\033[32mTEST\033[0m` text** — assumption fails, pivot to bold-only markdown (no ANSI)
- **(c) Stripped** — assumption fails, pivot to bold-only

**Suggested pivot if assumption fails** (per Codex recommendation): keep Option B's visual intent via existing markdown bold semantics — `**a)**` already bolds the letter in terminals that render markdown. Lose color, keep scanability. No ANSI codes in the spec at all.

### B2 (architectural · blocker · class:architectural)
**Literal `\033` is not an ANSI escape byte.** Codex finding #2: in Markdown source, `\033[32m` is six visible characters unless something decodes them through `printf '%b'` or shell-like escape interpretation. The spec assumes "Claude faithfully emits them" produces color — faithful emission would display the literal six characters. To actually color, the output bytes must contain `0x1b` followed by `[32m`.

This compounds B1 — even if Claude Code DID render ANSI in model output, the model would need to emit raw ESC bytes (not the printable `\033[32m` shown in the spec). The model's text channel does not have a straightforward way to emit 0x1b. The spec's design as written is structurally non-functional.

### B3 (contract · blocker · class:contract)
**AC4 grep regex misses real blocks and depends on non-portable grep.** Convergent finding from requirements and codex:
- `^- \*\*[a-z]\) ` only catches unindented bold-bullet lines. Misses indented bullets inside fenced templates (`commands/spec-review.md:245`, `commands/blueprint.md:217`, `commands/check.md:314`), non-bold letter choices (`commands/spec.md:38`), parenthesized options (`commands/spec.md:167`), and lettered-choice blocks in `commands/build.md` + `commands/kickoff.md`.
- `\033` is a Python/C escape, not a regex escape. macOS BSD grep does not expand `\x1b` the same way GNU grep does. The test needs `grep -P` (Perl regex) or `awk 'BEGIN{esc=sprintf("%c",27)}'` or `perl -ne` — none of which the spec specifies.
- AC4 says "immediately preceded within same line" but the regex as written is prose, not a literal grep command. A build agent could implement a vacuous check that passes trivially.

Even if B1/B2 are resolved, AC4 must be rewritten with literal portable test commands.

## Important But Non-Blocking

### I1 (scope · major · class:scope-cuts)
**Scope contradicts itself on file enumeration.** Convergent finding from scope and codex:
- Out-of-scope says "non-pipeline command files UNLESS they contain a lettered-choice block."
- Integration list names only `spec.md`, `spec-review.md`, `blueprint.md`, `check.md`.
- But `commands/kickoff.md` (domain-agent selector, ~3 lines) and `commands/build.md` have lettered-choice blocks — so they're in scope by the spec's own rule, but missing from the Integration list.

Fix: either explicitly add `kickoff.md` + `build.md` to the Integration list, or carve them out with a rationale. State the Integration list as "lower bound — grep discovery during build is the binding contract" (per OQ1).

### I2 (contract · major · class:contract)
**AC4's approval-prompt clause has no structural anchor.** Requirements finding: "Every approval prompt's interrogative line is preceded by `\033[32m`" — but there's no defined token like `**Q[N]` to anchor against. Unimplementable without guessing what counts as an "interrogative line."

Fix: anchor on a literal grep-able pattern (`^Approve to proceed`, `^Use this roster`, etc.) OR retire the approval-prompt clause entirely and let AC1+AC3 cover it.

### I3 (tests · major · class:tests)
**AC7's "no NEW ANSI codes outside lettered blocks" is over-broad.** Both requirements and codex flagged this. Without a baseline snapshot of existing ANSI usage (currently: zero), any future legitimate colored output anywhere in `commands/` would trip the test. Discourages useful changes.

Fix: retire AC7 OR convert it to a snapshot-diff test that compares against a checked-in baseline file enumerating allowed contexts.

### I4 (tests · minor · class:tests)
**AC4 doesn't test the blank-line rule from Option B.** Gaps finding: Option B specifies "blank line between question and options" but no AC asserts the blank line is present. Drift risk if a future edit collapses the spacing.

Fix: add a one-line check that asserts a blank line follows every `**Q[N]` line.
